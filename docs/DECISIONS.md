# DECISIONS.md — 架构决策记录 (ADR)

> 本文档记录项目中的重要设计决策及其原因。
> 格式：日期 | 决策 | 原因 | 替代方案 | 影响

---

## 2026-07-15 | 使用 UUID v4 作为主键

**决策**：所有数据库表使用 UUID v4 作为主键，不使用自增整数 ID。

**原因**：
1. 分布式友好：多实例写入不会冲突
2. 安全性：不暴露记录数量和创建顺序
3. 可离线生成：不需要数据库往返获取 ID

**替代方案**：
- 自增 ID：简单但暴露记录数，分布式需额外处理
- ULID：时间排序但兼容性不如 UUID

**影响**：
- 索引略大于整数 ID
- 需要 `uuid-ossp` 扩展
- URL 中使用 slug 而非 UUID

---

## 2026-07-15 | 使用异步 SQLAlchemy

**决策**：使用 SQLAlchemy 2.x 的异步模式 (`asyncpg` 驱动)。

**原因**：
1. FastAPI 本身就是异步框架，同步 ORM 会成为瓶颈
2. 高并发场景下异步 I/O 显著提升吞吐量
3. SQLAlchemy 2.x 对异步的支持已经成熟

**替代方案**：
- Tortoise ORM：更轻量但生态不如 SQLAlchemy
- 同步 SQLAlchemy + 线程池：简单但性能差

**影响**：
- 所有数据库操作必须 `await`
- Session 管理需使用 `async_sessionmaker`
- Alembic 迁移使用 `run_async_migrations`

---

## 2026-07-15 | MinIO 自托管 vs 云存储

**决策**：使用 MinIO 自托管对象存储，兼容 S3 API。

**原因**：
1. 数据自主可控
2. 开发/测试环境零成本
3. S3 兼容：未来可无缝迁移到 AWS S3/Cloudflare R2

**替代方案**：
- 直接使用 AWS S3：成本高，开发不便
- 本地文件系统：无法水平扩展

**影响**：
- 需要维护 MinIO 服务
- 生产环境需配置持久化存储和备份

---

## 2026-07-15 | Next.js App Router vs Pages Router

**决策**：使用 App Router（Next.js 14）。

**原因**：
1. 官方推荐的最新路由方案
2. Server Components 减少客户端 JS
3. 更好的布局嵌套和路由组织
4. SEO 友好的 SSR/SSG

**替代方案**：
- Pages Router：成熟但官方不再重点发展

**影响**：
- 需学习新范式和 RSC 模式
- 组件默认服务端渲染

---

## 2026-07-15 | 软删除策略

**决策**：所有数据使用软删除（`deleted_at` 字段）。

**原因**：
1. 数据可恢复：防止误删
2. 审计需求：保留数据历史
3. 用户合规：可能的数据删除请求

**替代方案**：
- 物理删除：简单但有数据丢失风险
- 归档表：复杂度和维护成本高

**影响**：
- 所有查询需添加 `WHERE deleted_at IS NULL`
- 需要考虑定期清理真正过期的数据

---

## 2026-07-15 | 模块开发顺序

**决策**：先开发无依赖的底层模块（users, categories, games），再开发有依赖的上层模块（resources, comments, ratings）。

**原因**：
1. 减少阻塞：底层模块完成后，上层可并行开发
2. 风险控制：核心模块先验证架构可行性

**影响**：
- MODULES.md 中的开发顺序即为依赖拓扑排序结果

## 2026-07-15 | 数据库表设计 — games 表 SEO 字段内置

**决策**：将 SEO 相关字段（seo_title, seo_keywords, seo_description）内置于 games 表，而非独立 SEO 表。

**原因**：
1. 减少 JOIN 查询次数
2. 每游戏仅有 3 个 SEO 字段，独立表收益不大
3. 简化 CRUD 操作

**替代方案**：
- 独立 `game_seo` 表：更规范化但查询复杂度高

**影响**：games 表增加 3 个 TEXT/VARCHAR 列

---

## 2026-07-15 | 数据库表设计 — categories 自引用实现层级

**决策**：categories 使用 `parent_id` 自引用实现无限层级分类。

**原因**：
1. 标准 SQL 方案，性能好
2. 支持任意深度的分类层级
3. 外键保证数据完整性（ON DELETE SET NULL）

**替代方案**：
- 邻接表 + 闭包表：查询快但写入复杂
- 物化路径：简单但难以维护

**影响**：删除父分类时子分类变为顶级（SET NULL）

---

## 2026-07-15 | 数据库表设计 — 下载渠道独立建模

**决策**：download_providers 作为独立表，download_resources 通过外键关联。

**原因**：
1. 渠道可独立管理（增删改、排序、启用/禁用）
2. 支持同一游戏多渠道下载
3. 渠道变更不影响游戏数据

**影响**：
- 删除渠道前需检查关联的下载资源（RESTRICT）
- 初始提供 8 个常见网盘渠道作为种子数据

---

## 2026-07-15 | 数据库表设计 — settings 表不使用软删除

**决策**：settings 表不使用 `deleted_at` 软删除。

**原因**：
1. 配置项数量极少且变更不频繁
2. 配置删除属于管理操作，不需要恢复
3. 简化查询（无需 `WHERE deleted_at IS NULL`）

**影响**：settings 是唯一不使用软删除的业务表

---

## 2026-07-15 | 数据库表设计 — game_tags 使用独立 UUID 主键

**决策**：game_tags 多对多关联表使用独立 UUID 主键，而非复合主键。

**原因**：
1. 与项目统一 UUID 主键策略一致
2. 独立主键便于未来扩展（如为关联添加元数据）
3. 唯一约束 `uq_game_tags_game_tag` 保证业务唯一性

**替代方案**：
- 复合主键 (game_id, tag_id)：更简单但不符合项目约定

**影响**：game_tags 表包含 id + game_id + tag_id 三列
